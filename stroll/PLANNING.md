# 하루 여행기 (Stroll) — 서비스 기획 참고 노트

> POC를 먼저 돌리며 쌓인 계획·실행·학습 기록. 정식 PRD를 쓸 때 이 문서를 토대로 삼는다.
> (작성 2026-06-24 / 프로토타입 위치: `stroll/`, 빌드 `scripts/build_stroll.py`·`export_reel.js`)

---

## 1. 한 줄 정의
GPS 태그된 **하루치 사진·영상**을 넣으면, **지도 위에 그날 동선이 애니메이션**되고 들른 곳마다 **폴라로이드(사진·영상) 슬라이드**가 뜨며, 통계 카드로 마무리되는 **9:16 인스타 릴스 "하루 여행기"**를 자동 생성한다.

## 2. 가치 가설 — 왜 쓰는가 (3축)
사용자(=초기엔 우리 자신)가 검증한 핵심 가치는 세 축이다.

1. **분위기 적합도** — 최소 입력(고른 미디어 + 짧은 브리프)으로 *여행 종류에 맞는 무드*를 살리는 것. 도심 하루 / 러닝 이벤트 / 문화(전시) / 교외·자연 / 해외여행이 각기 다른 느낌이어야 한다. (같은 엔진으로 시드니 하버 산책 vs Hoka 레이스 회고 vs OneFineDay 전시 데이가 전혀 다르게 나옴 = 검증됨.)
2. **공유 욕구** — "올리고 싶게" 만드는 것 = **화질(1080p 선명함)** + **영상 디자인 요소**(오프닝 훅, 키네틱 칩, 폴라로이드, 통계, 전환, 워터마크, CTA). "그냥 슬라이드쇼"와의 차이 = 프로덕션 밸류.
3. **입력 최소화 / 자동 큐레이션** — 사용자는 사진첩 정리를 귀찮아한다. **좋은 사진·영상을 자동 선별**(흐림·중복·안내판 제외, 베스트 추림)해 주는 것 자체가 핵심 가치. 이상적 입력 = "하루치 미디어 폴더 + 1줄 설명".

**Moat:** 지도가 아니다(구글도 지도는 한다). **① 최소 입력 → ② 종류에 맞는 분위기 → ③ 공유하고 싶은 완성도**, 이 셋을 동시에 주는 것.

## 3. 입력 / 의미의 분리 (설계 원칙)
- **GPS·EXIF = 뼈대(자동):** 어디·언제·날짜·대략의 동선·이동거리.
- **사람/비전 = 의미(브리프):** 무슨 이벤트인지, 교통수단, 지명, 제목, 톤, 대표 컷.
- → 의미는 **코드가 아니라 데이터(`brief.json`)**로 분리. 새 하루 = 사진 폴더에 brief 한 장이면 끝, 코드는 안 건드린다.

## 4. 동작 방식 (파이프라인)
```
사진/영상(EXIF GPS·시간)
  → 클러스터링(정류장)
  → 역지오코딩(지명) + landmarks(좌표 태깅으로 교정)
  → 이동수단 추정/지정
  → 경로 스냅 (Google Directions: 보도/선로/항로/트램 실제 노선)
  → 영상 6초 클립 변환
  → data.json
  → 웹 렌더(Leaflet/Carto, 마커 애니메이션 + 폴라로이드 + 훅/아웃트로)
  → puppeteer 헤드리스 Chrome 녹화 → ffmpeg → 1080×1920 mp4
```

## 5. brief.json — 데이터 주도의 핵심
폴더별 `<src>/brief.json`로 주입(없으면 순수 자동). 주요 필드:
- `title` / `date`(없으면 EXIF 자동) / `copy{}`(훅·인트로·아웃트로 문구, EN+KR)
- `landmarks[]` = **좌표로 지점 이름 태깅** `{name,lat,lon,r,feature}` (역지오코딩이 못 잡는 의미 부여)
- `force_mode` (전 구간 한 모드 고정, 예: 러닝) / `mode_overrides{}` (구간별 모드 지정)
- `collapse_dupes` (연속 같은 지명 병합 = 지그재그 제거)
- `hero_photos[]` / `hero_cutout`(false면 컷아웃·테두리 없이 보정 사진) / `hero_mask`(컷아웃에 같이 잡힌 타인 제거)
- `feature`(landmark별) = 그 stop은 여러 컷 느긋하게(시간축 샘플), 아니면 베스트 1컷 빠르게

## 7. 핵심 학습 (분야별)
- **지오코딩:** 도심은 자동이 "Sydney/Civic Plaza/The Rocks"로 뭉뚱그림 → 좌표 landmark + 비전으로 의미 부여가 필요.
- **이동수단 감지:** GPS 속도만으론 오분류(러닝→페리, 트램→버스로 뜸). 경로 단독은 애매. **비전(사진 속 정류장·Opal 단말기·차량)이 가장 신뢰 가능한 모드 감지기.** 모드를 알면 경로는 `transit_mode`로 실제 노선을 그린다.
- **릴스 리텐션:** 첫 1~3초에 차별점(움직이는 지도)이 보여야 함 → 인트로를 ~2초로 압축하고 바로 지도로.
- **화질:** 헤드리스 screencast가 CSS px로 캡처(deviceScaleFactor 무시)하고 어둡게 찍힘 → ① 네이티브 1080으로 녹화 ② 훅 구간만 ffmpeg eq 보정(전역 감마는 밝은 지도가 워시아웃) ③ 모션은 H.264 압축에 뭉개짐 → 히어로·텍스트는 정적으로.
- **연출 페이싱:** stop 성격별로 컷 수·체류시간을 다르게(러닝=1컷 빠르게 / 출발·결승·기념=여러 컷 느긋, 시간축 고르게 샘플해 도착~후식까지 포함). 영상은 끝까지 재생(클립 6초@2× = 3초 dwell).
- **큐레이션:** 검수 안 한 사진을 다 넣으면 폴라로이드가 안 예쁨 → 자동 선별(비전 triage)이 옵션이 아니라 핵심 기능.
- **렌더 선호(확정):** 폴라로이드 장소명은 **영어**, 프레임은 크게(1.2×), 이미지 크롭은 센터 기본(전역 top 정렬은 어색).
- **이동수단 = 7종 + 모드별 마커(2026-06-25):** 도보🚶/러닝🏃/트램🚊/기차🚆/페리⛴️/**차🚗**/**비행✈️**. 모드마다 ① 마커 SVG(app.js `travelerIcon`, 서쪽 진행 시 좌우 반전) ② 경로 geometry 규칙(차=driving 스냅, 기차/트램=transit 실노선, 비행=arc 베지어) ③ 한글 라벨(build_stroll `LABEL`)이 한 세트. **교훈:** 새 이동수단 = "마커 + geometry + 라벨" 세 곳만 추가하면 끝(설계가 모드 단위로 잘 쪼개져 있음).
- **거리 집계:** 레그별 실제 geometry 길이(haversine)를 **모드별로 합산** → `data.modes[{mode,label,km,count}]` + `total_km`. 훅·아웃트로 통계에 그대로 노출. 비행은 arc 길이라 대권거리보다 약간 길지만 표시용으론 충분(예: 인천→시드니 8,417km).
- **대륙간 비행 카메라(2026-06-25):** 비행 레그는 마커를 따라 panTo 하면 베이지 땅만 보임 → **`flyToBounds(arc 전체)`로 출발·도착 대륙을 한 프레임에 고정**하고 비행 중엔 panTo를 끈다(`travelLeg`에서 flight 제외). 그래야 "대양을 가로지르는" 장면이 산다. 줌은 확 빼고(world zoom), 마커는 키우고, 항로 점선이 도착지까지 미리 보이게.

## 9. 기술 함정 메모 (재현 시 주의)
- screencast 캡처는 CSS 뷰포트 px(=네이티브 1080으로 녹화해야 선명).
- 1080p 영상 생성(Higgsfield)은 유료 플랜 필요.
- 히어로 컷아웃 캐시는 **소스별 파일명**으로(데이터셋 바뀌면 재생성). rembg human_seg는 군중에서 엉킴 → 단독 피사체 컷 권장, 같이 잡힌 타인은 좌표 마스크로 제거.
- 변수 섀도잉 주의: 경로 루프의 `A,B=stops`가 브리프 변수를 덮어쓴 버그 있었음(→ `BRIEF`로 분리).
